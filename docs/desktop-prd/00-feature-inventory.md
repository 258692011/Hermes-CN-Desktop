# 桌面端 PRD · 00 · 原型功能盘点

> **用途**：写桌面端 PRD 之前的输入文档。把 `hermes-cn-ui-prototypes/` 36 个高保真原型页"画了什么"逐页提炼一遍，作为 PRD 取舍的素材。
>
> **来源**：`hermes-cn-ui-prototypes/src/pages/[theme]/[lang]/*.astro`（编号 01–21，含 a/b 子变体）
>
> **当前状态**：仅为「能力盘点」，**不代表桌面端最终 scope**。PRD 阶段还需要做以下决策：哪些 a/b 变体取一个、哪些原型功能延后、哪些原型仅 Web 用不带进桌面。

## 1. 工作台（Workbench）

### `01-workbench` — 主仪表盘
- **布局**：顶栏 + 左侧栏（会话列表 + 项目快捷）+ 主区 + 右侧栏（活动 Feed）
- **核心区块**：
  - Hero 问候
  - Compose 编辑器（直接写任务）
  - **健康检查网格**（7 指标卡：网关 / Token / Home / 模型 / Skills / MCP / 工作区）
  - 进行中任务卡
  - 配方/模板入口
  - 跨项目活动 Feed
- **依赖**：进行中任务 API、7 项健康探针、会话历史 Feed、模板/Skills 列表

### `01b-workbench-new` — 首次启动空状态
- 欢迎 + 入门 3 步检查清单 + 零状态卡片 + Preflight 清单
- 已具备工作区/模型/审批策略选择
- **依赖**：网关可达性、Token 往返、工作区列表

### `02-new-task` — 新建任务编辑器
- Composer（提示词 + 附件 + `@file` / `/skill` 标签）
- 工作区 / 模型 / 审批策略选择
- 6 张配方模板卡
- 右侧元信息：工作区、模型能力、消费预估
- **依赖**：Token 估算、工作区/模型/Skills/模板 API

### `02b-new-task-filled` — 草稿待发送
- 同 02 但所有字段已填、Preflight 8 项全通过
- 强调 "ready to send"、`⌘↵` 快捷
- **依赖**：Preflight API（工作区有效、Skills 就绪、模型可达、Token 未过期、审批策略）

### 任务详情组（03–06 + 各自 b 变体）
共用三栏布局：**左侧栏 + 中间 Chat Timeline + 右侧工作区面板**。中间 Timeline 元素相同（用户/助手轮次、工具调用块、思考块、批准卡），仅右侧 Tab 切换。`b` 变体一律是「折叠右侧工作区面板、专注 Chat」的聚焦模式。

| 页面 | 右侧 Tab | 关键控件 | 后端依赖 |
|------|----------|----------|----------|
| `03-task-detail-files` | 文件树 + Diff 高亮 | 搜索/过滤、面包屑、修改标记 | 文件树 API、文件内容、Diff |
| `04-task-detail-artifacts` | 产物卡片/预览 | 下载、复制 | 产物清单 + 预览 |
| `05-task-detail-terminal` | 终端单色输出 | 停止 / 刷新 | 终端 SSE、进程控制 |
| `06-task-detail-logs` | 日志流 | Level 过滤、搜索 | 日志 SSE、日志查询 |

> 这一组是**整个桌面端最重 SSE / 流式依赖**的页面，4 类右侧面板都假设有实时流。

## 2. 项目（Projects）

### `07-projects` — 项目列表
- 网格 / 列表视图切换
- 每张项目卡：名称、颜色、会话数、最近活动时间
- 侧栏：跨项目进行中会话
- **依赖**：项目列表 + 元数据（会话数、最后活动）

### `08-project-detail` — 项目详情仪表盘
- 项目概览卡 + **Git 分支/改动** + 文件树片段 + 项目内活动 Feed
- 侧栏：项目内活动会话、今日统计
- **依赖**：Git 状态 API、项目文件树、项目内会话 Feed

## 3. 能力（Capabilities）

侧栏统一含**子页导航：Skills / MCP / Cron**（注意 Cron 在这里也作为入口出现一次，但顶导里 Cron 属于 Automation——存在导航交叉，PRD 要消解）。

### `09-skills` — Skills 详情视图
- Tab 过滤：全部 / 系统 / 我的 / 插件
- 表格：技能名、触发器、来源、使用次数、启用开关
- 详情面板
- **依赖**：Skills 清单、元数据、启用/禁用 API

### `09a-skills` — Skills 画廊（网格卡片版）
- 同 09，换成视觉网格卡

### `09b-skills-mine` — 我的 Skills 编辑器
- 左：我的 Skills 列表；右：YAML/JSON 配置编辑器 + 测试器 + 保存
- **依赖**：Skill 配置读写、Skill 测试运行

### `10-mcp` — MCP 服务器管理
- 服务器列表 + 在线状态
- 每个服务器的工具列表、连接状态、错误诊断
- 重连、添加新服务器
- **依赖**：MCP 服务器列表、实时在线状态、工具清单、连接控制

## 4. 自动化（Automation）

### `11-cron` — 例程管理
- 左：例程编辑器（cron 表达式、提示词、参数）
- 右：时间轴/日历，展示执行历史 + 下次运行
- 侧栏子页：**例程 / 触发器 / Webhooks**（原型只画了例程一页）
- **依赖**：Cron 读写、执行历史、下次预测、手动触发/测试

## 5. 可观测（Observability）

### `12-logs` — 全局日志聚合
- 多来源（应用 / 系统 / 工具）日志统一
- 时间范围 + Level + 来源过滤、搜索
- 行可展开详情、可跳到关联会话
- **依赖**：日志查询 + 实时 SSE

### `13-health` — 系统健康
- Hero：当前状态 + 问题数
- 7 项健康检查卡（同 workbench 的健康网格）+ 每项的修复 hint
- 手动刷新
- **依赖**：7 个探针 + smart hints

### `14-models` — 模型供应商
- 左：供应商列表 + 连接状态
- 右：选中供应商的模型、能力、消费、配额
- 星标设置默认、编辑凭证、测试连接
- **依赖**：供应商配置、模型能力清单、实时消费/配额、凭证验证

## 6. 模型（顶导第 6 项）

入口直接是 `14-models`，与可观测下的 models 是**同一页**。语义上 PRD 要么把 models 从可观测里去掉、要么把顶导第 6 项去掉，避免双入口。

## 辅助页

### 登录 / 凭证

- **`16-login`** — Onboarding 3 步：网关地址 + Token 粘贴 + Preflight。**这是桌面端首次启动的第一屏**。
- **`15b-settings-credentials`** — 凭证管理（LLM / MCP / 第三方），含过期告警 + 测试有效性。

### 设置

- **`15-settings`** — 工作台偏好 + 审批策略 + 默认模型 + 通知，左导菜单 + 右内容。

### 档案

- **`20-profiles`** — 个人档案查看与编辑（用户名、时区、语言、邮件、工作室标签）
- **`20a-profiles-cards`** — 档案卡片网格，**快速切换**
- **`20b-profiles-detail-pane`** — 列表 + 详情双栏，管理向

> "工作室 / 档案" 是原型里**新引入的概念**，README/CLAUDE.md 都没提。PRD 要决定：是 Hermes 后端已有的 profile（即 `hermes profile switch`）的 UI 化，还是新的"多身份"抽象？

### 全局交互

- **`17-search-modal`** — `⌘K` 命令面板。过滤芯片：全部 / 会话 / 文件 / 命令 / Skills / 项目。Modal scrim 冻结背景。
- **`19-conversation-history`** — 全部会话表格视图（搜索、排序、跳转），与工作台侧栏的会话列表是同源数据但全量视图。

### 日报

- **`21-daily`** — 出版物风格的日报（Masthead + Lede + 数据特稿）。
- **`21a-daily-mobile`** — 同 21 的响应式移动版。**桌面端是否需要？** PRD 要决策。

### 品牌页

- **`18-logo`** — 设计系统展示，**非用户功能页**，仅内部参考。桌面端打包应排除。

## 跨页设计模式（重复出现）

1. **三栏 + Chat Timeline**：04 大组任务详情共用，是桌面端最复杂的页面，强依赖实时流。
2. **侧栏会话上下文**：除任务详情外，几乎所有顶导页面侧栏都保留「进行中会话 / 项目快捷」。
3. **列表 / 网格 / 卡片三视图**：09 ↔ 09a、20 ↔ 20a ↔ 20b 各自给了多种展示形态。PRD 要么"全做、用户切"，要么"选一种作为默认 + 其它砍掉"。
4. **Preflight 模式**：01b / 02b / 16 都有一份「N 项检查、每项有状态徽标」的清单组件。强烈建议抽公共组件。
5. **健康检查 7 项**：workbench `01` 和 health `13` 用了同一份探针数据，两处都展示。

## PRD 阶段需要明确决策的清单

把原型变成产品需要回答的开放问题，按风险高到低：

1. **导航交叉**：Cron 同时出现在「能力」侧栏和「自动化」顶导；models 同时在「可观测」侧栏和顶导第 6 项。要么各砍一处，要么明确主入口 + 次入口语义。
2. **"工作室 / 档案"概念**：原型里 20/20a/20b 给了一套档案体系。是 Hermes profile 的可视化，还是新增抽象？这直接决定后端是否要扩。
3. **"项目"vs"工作区"**：原型有 07/08 项目页，但 Hermes 后端是 workspace 概念。两者是同一回事还是项目是工作区之上的元概念？
4. **a/b 变体处理**：
   - 03b–06b 折叠态 → 收敛为「右侧面板可隐藏的切换」即可，不需要两套路由
   - 09 vs 09a 视图切换 → 是用户偏好还是只保留一种？
   - 20 vs 20a vs 20b → 一定要砍到一种
5. **日报功能**：21/21a 是从零开始要做的大特性。桌面端 v1 是否包含？移动版 21a 与桌面端无关，但提示"日报内容也要 Web 可访问"。
6. **触发器 / Webhooks**：原型只画了 Cron 例程，子页导航暗示后续要做触发器和 Webhooks 但原型未覆盖。v1 是否纳入？
7. **桌面端独有 vs Web 共用**：原型本身是 Web，但桌面端需要原生窗口、文件对话框、托盘、自动更新（参考 README）。哪些功能因桌面端 affordance 需要重新设计（如：拖拽附件、原生菜单、深链接、本地通知）？
8. **18-logo 品牌页**：明确不打包进桌面端产物。
9. **首次体验链路**：`16-login → 01b-workbench-new → 02-new-task → 03 任务详情`。要不要在桌面端做强引导（步进 wizard）？还是只用 Preflight 清单提示？
